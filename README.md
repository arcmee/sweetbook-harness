# sweetbook-harness

`sweetbook-harness`는 Sweetbook 프로젝트의 하네스 엔지니어링 문서를 관리하는 저장소입니다.

이 저장소는 애플리케이션 소스코드와 분리하여, 하네스 엔지니어링에 필요한 제품 흐름 정의, 작업 워크플로우, 에이전트 역할 정의, 연동 메모를 체계적으로 관리하기 위해 운영됩니다.  
즉 실제 기능 구현 자체보다, 구현을 안정적으로 진행하기 위한 기준 문서와 운영 문서를 한곳에 모아두는 것을 목적으로 합니다.

## Repository Purpose

이 저장소는 다음 목적을 위해 사용됩니다.

- 하네스 엔지니어링에 필요한 기준 문서를 코드 저장소와 분리하여 관리
- 작업 단계별 역할, 협업 규칙, 자동화 흐름을 명시적으로 정의
- 구현 전후에 참고할 수 있는 도메인 규칙과 API 메모를 정리
- 제품 흐름, 테스트 흐름, 작업 분업 구조를 반복 가능한 방식으로 유지

## What Harness Engineering Means Here

이 저장소에서 말하는 하네스 엔지니어링은 다음을 포함합니다.

- 제품 흐름을 구현 가능한 단계로 정리하는 일
- 에이전트별 책임과 작업 순서를 명확히 정의하는 일
- 구현, 테스트, 리뷰, PR 정리까지 이어지는 반복 가능한 작업 체계를 만드는 일
- 실제 애플리케이션 저장소에서 동일한 기준으로 작업할 수 있도록 문서 기반 운영 구조를 만드는 일

즉 이 저장소는 단순 문서 보관소가 아니라, Sweetbook 프로젝트를 일관된 방식으로 설계하고 구현하기 위한 작업 하네스를 제공하는 역할을 합니다.

## Directory Structure

### `agents/`
작업 역할별 문서를 포함합니다.

예:
- Planner
- Implement
- Review
- Security
- Git
- PR

### `workflow/`
작업 순서와 협업 절차 문서를 포함합니다.

예:
- AI workflow
- Subagent orchestration

## Key Documents

### `group-based-product-flow-spec.md`
Groupictures의 그룹 기반 제품 흐름과 주요 사용자 시나리오를 정리한 문서입니다.

### `project-foundation.md`
프로젝트의 기본 방향과 기반 구조를 정리한 문서입니다.

### `sweetbook-api-notes.md`
SweetBook 연동에 필요한 메모와 API 관련 참고사항을 정리한 문서입니다.

## Usage

이 저장소는 다음과 같은 상황에서 기준 문서 저장소로 활용됩니다.

- 하네스 엔지니어링 기준을 먼저 정리할 때
- 제품 흐름과 화면 구조를 재정리할 때
- 기능 구현 전에 기준 시나리오를 확인할 때
- 작업 단계를 planner, implement, review 흐름으로 나누어 진행할 때
- SweetBook 연동 또는 도메인 규칙을 다시 검토할 때

## Current Scope

현재 저장소는 아래 범위를 중심으로 구성되어 있습니다.

- 하네스 엔지니어링 기준 문서
- 그룹 기반 제품 흐름 정의
- 작업용 에이전트 역할 문서
- 협업 및 자동화 워크플로우
- SweetBook 관련 메모와 참고 자료

## Notes

- 실제 애플리케이션 코드는 별도 저장소에서 관리됩니다.
- 본 저장소는 실행 가능한 애플리케이션이 아니라, 하네스 엔지니어링과 프로젝트 운영 기준을 위한 문서 저장소입니다.
